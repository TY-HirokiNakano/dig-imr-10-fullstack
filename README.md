# マラソン練習アプリ

## 概要

このアプリは、ハーフマラソンの目標タイムを入力することでジョギングでの推奨ペース、目標のレベルを表示します。練習のモチベーションを高めるねらいがあります。

## 事前準備

以下がインストールされていることを確認してください。

- [ ] [Node.js](https://nodejs.org/ja/) (動作確認のバージョン24.14.1)
- [ ] [PostgreSQL](https://www.postgresql.org/) (動作確認のバージョン18.3)

## アプリ起動までの手順

依存関係のインストール

```sh
npm install
```

DBのセットアップ

```sh
npm run db:migrate
npm run db:seed
```

アプリの起動

```sh
npm run dev
```

HTTP とは Hypertext Transfer Protocol（ハイパーテキスト転送プロトコル） の略で、コンピュータ同士がインターネット上で通信するための標準的な仕組みであり、クライアントとサーバーが情報をやり取りするための方法です。

- クライアント：リクエストを送るプログラム
- サーバー：リクエストを受け取り、レスポンスを返すプログラム

普段私たちが使っている Web ブラウザも HTTP クライアントの一種です。ただし、HTTP リクエストを送る方法はブラウザだけではありません。この課題で使用する Postman も、HTTP リクエストをテストしたり、サーバーからのレスポンスを確認したりするための HTTP クライアントと言えます。  
この課題では、Express サーバーに対して Postman を使ってリクエストを送信していきます。

## セットアップ

### 1. Express パッケージをインストールする

Express は HTTP サーバーの作成や URL エンドポイント（リクエストを受け取る先の URL）を簡単に定義することができるパッケージです。  
Express を使うには、まず npm（Node Package Manager）というパッケージマネージャーを使ってインストールする必要があります。

- [ ] Express パッケージをインストールする： `npm install express`

Express のインストールが完了すると、`package.json` ファイルの "dependencies" の欄に `express` が追加されているはずです。 `package.json` で npm でインストールしたすべてのパッケージを管理します。では、これらのパッケージは実際どこに保存されているのでしょうか？  
それらはプロジェクト内の `node_modules` というローカル（＝ GitHub ではなく手元のコンピュータ中の）ディレクトリに保存されます。このディレクトリには、アプリケーションを動かすために必要なすべてのパッケージのコードが含まれています。

ただし、これらのファイルは自分で書いたり編集したりするものではないため `node_modules` ディレクトリは git リポジトリに含めるべきではありません。

### 2. `.gitignore` ファイルを作成する

特定のファイルやディレクトリを git に含めないようにする（つまり git 管理下から外したい場合）には、 `.gitignore` というファイルを使います。

- [ ] プロジェクトのトップディレクトリに `.gitignore` という名前のファイルを作成する
- [ ] `.gitignore` の1行目に `node_modules` と記述する
- [ ] `git status` を実行して `node_modules` が無視されていることを確認する（ステータス表示に node_modules が含まれていなければOKです）

`node_modules` のほかにも、ログファイル、キャッシュ、エディタの設定ファイル（例：`.vscode`）なども git 管理から除外すべきです。基本的には、自分で書いたファイルだけをバージョン管理するようにしましょう。

### 3. Express サーバーを作成・起動する

Express パッケージをインストールしたので、コード内でインポートして使用できるようにしましょう。

- [ ] `server.js` ファイルの先頭で `require` 文を使って `Express` をインポートする： `const express = require("express");`

次に、サーバーを作成します。`express()` を実行するとサーバーアプリケーションが生成されるので、その結果を変数に代入します。

- [ ] Express アプリケーションを作成する： `const app = express();`

これでサーバーの準備はできましたが、どうやって動かせばよいのでしょうか？
それには、サーバーを実行しクライアントからのリクエストを受け付けられる状態（このことを listen と言います）にする必要があります。

Express には `listen` という関数があり、それを使うことでサーバーを起動しクライアントからのリクエストを受け付けられるようになります。

🔎 [`app.listen` のドキュメント](https://expressjs.com/ja/4x/api.html#app.listen) を参考にしてください。

- [ ] まず `PORT` という定数を定義し、値として `3000` を代入する： `const PORT = 3000;`  
       ポートは、クライアントのリクエストが「どこに届くか」を指定するための “郵便受け番号” のようなものです。1 つのプロセスで同じポートは 1 回しか使えないため、すでに使われているポートは利用できない場合があります。
- [ ] `app` に対して `listen` を呼び出し、引数として `PORT` とコールバック関数を渡す  
       そのコールバック関数では、`console.log` を使って "Server is running" と表示するようにしてください。

  ```
  app.listen(PORT, () => {
    console.log("Server is running");
  })
  ```

- [ ] コマンドラインからサーバーを起動する： `node server.js`

### 4. Express サーバーにアクセスする

サーバーが正しく起動していれば、ターミナルに "Server is running" というメッセージが表示されているはずです。

- [ ] Postman を開き、新しいタブで次のURLを入力する： `http://localhost:3000`
- [ ] HTTPメソッドが「GET」になっていることを確認し、「Send」をクリックする

レスポンスは Postman の画面下部で確認できます。
`Body` タブの中に、「Cannot GET /」というメッセージを含む HTML が表示されていると思います。

これはどういう意味でしょうか？

`GET` とは **HTTP メソッド** のひとつで、クライアントがサーバーからデータを取得したいときに使います。そして、Web サーバーへのアクセス先は `/` というパスです。しかし、サーバー側には `/` パスにリクエストが来たときにどんなレスポンスを返すかの処理がまだ書かれていません。  
つまり 「Cannot GET /」 というメッセージは、サーバーが「クライアントから `GET` メソッドで `/` のパスに HTTP リクエストが送られてきたけど、それに応答する方法がわからないよ！」と言っているのです。要するに、リクエスト自体は届いているけれど、サーバー側にその処理方法が定義されていないためエラーとして返されているということです。

### 5. Express サーバーにエンドポイントを作成する

ではこのエラーを解決していきましょう!

- [ ] `app.get` を使って、パス `/` に対する GET リクエストの処理を定義する
  - [ ] 第一引数に、パス `/` を指定する
  - [ ] 第二引数に、リクエストとレスポンスを引数に取るコールバック関数を指定する：  
         `app.get("/", (req, res) => res.send("I've received your request!"));`

エンドポイント、つまりリクエストを受け取る URL 側のコールバック関数は、次のような形式で記載します。

```javascript
(req, res) => {
  // ここに処理を記述する
};
```

第一引数は、 [Express リクエストオブジェクト](https://expressjs.com/ja/api.html#req) で、クライアントから送られてくる HTTP リクエストに関するデータや機能が含まれています。通常、request を短縮した `req` という名前が使われます。

第二引数は、[Express レスポンスオブジェクト](https://expressjs.com/ja/api.html#res) で、サーバーからクライアントに送る HTTP レスポンスに関するデータや機能が含まれています。通常、request を短縮した `req` という名前が使われます。

このコールバック関数の中で、サーバーがクライアントからのリクエストをどのように処理し、クライアントにレスポンスを送信すべきかを定義します。

これでサーバーは `http://localhost:3000/` にリクエストが送られた時に、どんなレスポンスをすれば良いか理解できる状態になりました。ですが、今の段階でブラウザでアクセスしてもまだ何も変化が見られないかもしれません。
なぜなら、サーバーのコードを変更した後は、変更を適用するためにサーバーを再起動する必要があるからです。

- [ ] ターミナルからサーバーを停止する
  - Mac の場合： `⌘ + C`
  - Windows または Linux の場合： `Ctrl + C`
- [ ] もう一度サーバーを起動する： `node server.js`
- [ ] そして、ブラウザで `http://localhost:3000` にアクセスする
  - 今度はブラウザに "I've received your request!" というメッセージが表示されるはずです。

### 6. nodemon パッケージをインストールし、起動コマンドを定義する

でも、コードを変更するたびに毎回サーバーを手動で停止したり起動したりするのって面倒ですよね？ありがたいことに、ファイルの変更を検知して自動でサーバーを再起動してくれる便利なパッケージがあります。それが [_nodemon_](https://www.npmjs.com/package/nodemon) です。

- [ ] nodemon を npm でインストールする： `npm install --save-dev nodemon`
  - `--save-dev` オプションをつけることで、このパッケージは開発時のみ必要なものだと npm に伝えています。

`nodemon` をインストールしたので、サーバーを起動するときに `node` の代わりに `nodemon` を使えるようになりました。でも、毎回コマンドを入力するのは面倒なので npm スクリプトとして定義しておきましょう。

- [ ] `package.json` ファイルに `"scripts"` というキーを追加し、その値にオブジェクトを指定する
- [ ] `scripts` オブジェクトの中に `"start"` というキーを追加し、値に `nodemon server.js` を指定する

`scripts` オブジェクトは以下のようになるはずです：

```json
"scripts": {
  "start": "nodemon server.js"
},
```

実は、 `package.json` の `"scripts"` オブジェクトに記述したものは、ターミナルからコマンドとして実行することができます。

```shell
npm run <ここにコマンド名を入力>
```

というわけで、次回からは以下のコマンドを実行するだけで自動再起動付きの開発サーバーが立ち上がるようになります。

```shell
npm run start
```

ちなみに、start は特別なスクリプト名なので、他のスクリプトとは違って `run` を省略することができます。つまり、 `npm run start` の代わりに `npm start` で実行することができます。なお、start 以外では `run` は省略できません。

コマンドラインで次のコマンドを実行してみましょう：

```shell
npm start
```

すると、これまで通りサーバーが起動しますが、先ほど `nodemon` を入れたので今度は `server.js` のコードを変更するたびに、自動でサーバーが再起動されるようになります。  
試しに `"I've received your request!"` というメッセージを別のメッセージに変更して、ブラウザで確認してみてください。

これで、開発用サーバーが動作するようになりました！  
ただし、コードにバグがあるとサーバーがクラッシュすることがあります。その場合は、手動でサーバーを再起動する必要がありますので注意してください。

## 基礎演習

引き続き `server.js` ファイルに下記のコードを実装していきましょう！

### エンドポイントの作成

- [ ] `/butterfly` というパスに GET エンドポイントを追加し、"Monarch" というテキストを返すようにします。
- [ ] 上記のパスに対して、Postman から GET リクエストを送って正しく応答されるか確認しましょう。
- [ ] `butterflies` という配列を定義します。配列の要素は以下です。（ちなみに、これらは蝶の名前です）
  - "Monarch", "Swallowtail", "Cleopatra", "Viceroy"
- [ ] `/butterflies` というパスに GET エンドポイントを追加し、以下のように `butterflies` 配列の内容を「カンマ区切りで、かつ最後に and をつけた文字列」で返すようにします。
  - "Monarch, Swallowtail, Cleopatra, and Viceroy"

### クエリパラメータに対応したエンドポイントの作成

クエリパラメータ（クエリ文字列）とは、クライアントからサーバーへ URL を通して値を送る方法のひとつです。URLの末尾に `?` をつけて、`/path?key=value` のような形式で指定します。

参考：

- [クエリパラメータ](https://en.wikipedia.org/wiki/Query_string)
- [Expressクエリパラメータ](https://expressjs.com/ja/api.html#req.query)

> [!WARNING]
> クエリパラメータで渡される値はすべて文字列として扱われます。
> 数値として使いたい場合は、**サーバー側でパース（変換）** する必要があります。
> **パスワードやクレジットカード番号などの機密情報は、絶対にURLのクエリパラメータとして送ってはいけません！**

- [ ] GET `/butterfly` エンドポイントを更新し、`index` というクエリパラメータを受け取れるようにしてください。
  - [ ] その `index` の位置にある `butterflies` 配列の要素をレスポンスとして返します。
- [ ] 指定されたインデックスが配列の範囲外（0 未満 または 配列の長さ以上）の場合、エラーメッセージ "Index is out of range" を返してください。
  - [ ] この場合、HTTP ステータスコードを 500（サーバーエラー） に設定してください。
  - **ヒント** : [Express response status](https://expressjs.com/ja/api.html#res.status) を使用します。
- [ ] `index` の値が数値でない場合、エラーメッセージ "Index must be a number" を返してください。
  - [ ] この場合、HTTP ステータスコードを 400（クライアントエラー） に設定してください。

### パスパラメータ（ルート・パラメータ）に対応したエンドポイントの作成

クエリパラメータ以外にも、パスの中に値を直接含める方法があります。

- [ ] [Express ルート・パラメータ](https://expressjs.com/ja/guide/routing.html#route-parameters) について読んでみましょう。
- [ ] `GET /butterfly` エンドポイントを、クエリ文字列ではなく名前付きのパスパラメータ（index）を使うようにリファクタリングしてください。
- [ ] 新しく GET エンドポイントを追加し、パスパラメータで渡された 2 つの整数の合計をテキスト形式でレスポンスとして返すようにします。
  - 例）`/2/plus/3` というパスにアクセスが来たら `"5"` という結果を返してください。

`/butterfly` のリクエスト URL を変えることで、返ってくる内容も変わることに注目してください。  
このようにリクエストごとに異なるレスポンスを返すしくみを **「動的コンテンツ（dynamic content）」** と呼びます。同じエンドポイントでも、パラメータによって返す内容を変えられるのが特徴です。

### 静的コンテンツを応答するための設定追加

サーバーはクライアントにテキストを返すだけでなく、もっといろいろなことができます。
その機能の 1 つが「ファイルの共有」です。たとえば、クライアントはサーバーに画像ファイルをリクエストすることもできます。

サーバーにおいて画像ファイルのように内容が変わらないファイルのことを、「静的ファイル（static file）」 や 「静的コンテンツ（static content）」 と呼びます。
なぜ **「静的」** と呼ばれるのでしょうか？それは、どんなリクエストの仕方をしても、サーバーから返される内容（ファイルの中身）が常に同じだからです。

このプロジェクトには `public` というディレクトリの中に画像ファイルが 1 つ入っています。しかし、今の状態で `/public/image.jpg` に GET リクエストを送っても画像は返ってきません。

静的コンテンツを扱えるようにするには、Express に設定を追加する必要があります。

- [ ] `server.js` の中で、`app.listen` の1行上に次のコードを追加してください：

  ```javascript
  app.use(express.static("./public"));
  ```

これで、次のようにリクエストを送れば画像ファイルが取得できるようになります：

```shell
GET `http://localhost:3000/image.jpg`
```

このURLは Postman でもブラウザでも使えます。ブラウザでアクセスすれば画像が表示されるはずです。
`express.static` 関数は、プロジェクト内の静的ファイルを置くためのディレクトリを指定し、クライアントからの静的ファイルのリクエストをどう処理するかを Express に伝えるためのものです。
